# Migrate agent-step 1.2.0 → 1.3.0

> Severity: additive. Applied by `/pull-library`; readable standalone.

## What changed

The library gains **library-coordinated channel handoff** for specialized agents — entirely
opt-in; nothing existing changes behavior:

- New module `handoff.ts` (+ `handoff.test.ts`): `HandoffSpec` / `HandoffOffTopicSpec` /
  `HandoffDelegateTarget`, `createHandoffNode(spec)` (graph node resolving a pending handoff:
  terminate envelope, or delegation to another LangGraph deployment over the Platform API with
  live token pass-through and a behavioral fallback to the envelope), `handoffRequested(state)`
  (edge predicate), `HANDOFF_ACTION` (`"request_handoff"`), `HANDOFF_NODE`
  (`"resolve_handoff"`), `HANDOFF_ACTION_DESCRIPTION`, `handoffParamsSchema`,
  `HANDBACK_SIGNALS` (reason → `handoff_type` signal: `off_topic` → `"OFF_TOPIC"`).
- `BuildAgentStepToolOptions` gains optional **`handoff?: HandoffSpec<T>`**. When provided, the
  runner auto-injects the reserved `request_handoff` action: a pure slot write, **sole-step**
  (`error: "handoff_must_be_sole_step"` refuses the whole batch otherwise), no prereqs, and
  allowed as the first step under input lockdown. The name is reserved ONLY while the opt is
  provided — **a tool that does not opt in keeps the right to its own `request_handoff`
  action** (the orchestrator/scaffold handoff mechanism names its outbound-routing action
  exactly that), so pulling 1.3.0 is safe for such projects with no rename.
- `state.ts` gains the library-managed **`handoff: HandoffRequest | null`** slot
  (`HandoffRequestSchema`, `{ reason: "off_topic", context }`), included in
  `agentStepStateSpec` / `agentStepZodShape` / `LibraryManagedSlots` — hosts that spread the
  fragments get it for free.
- Final-message contract (emitted by `createHandoffNode`): terminate / delegate-failure
  fallback → the OFF_TOPIC handback `additional_kwargs`
  (`is_handoff: true, handoff_type: "OFF_TOPIC", handoff_reason: <the customer's request>,
  handoff_metadata: { service_type, success_message }`); delegate success → **not a handoff**
  (conversation kept) — informational `{ delegated_to }` only. Streaming clients of a
  handoff-enabled agent need `stream_mode: ["messages-tuple", "custom"]` for the
  control-plane events (`handoff`, `delegated_token`, `handoff_complete`,
  `delegated_restart`).

## Why

Specialized agents (see `streaming-and-channel-contract.md` `<agent_roles>`) need their
off-topic plays as infrastructure, not per-project code: hand the conversation back to the
orchestrator (`OFF_TOPIC` handback), or route a single off-topic turn to a general agent and
keep the conversation (delegate mode). The detection stays prompt-driven; the runner stays
pure (the slot write); all I/O lives in the graph node.

## Library replacement

The vendored `src/agent-step/*` is replaced wholesale by `/pull-library` (verbatim copy from
the toolkit) — including the NEW `handoff.ts` and `handoff.test.ts` files. The test suite
grows 65 → 76.

## Manual follow-ups

Adopting the feature itself is a design decision, not a transform (skip entirely if this agent
does not hand off):

1. **Env→spec module** (e.g. `src/handoff-env.ts`): fail-fast config —
   `HANDOFF_OFF_TOPIC_MODE` (`terminate` | `delegate`) + `HANDOFF_TERMINATE_MESSAGE` always;
   `HANDOFF_DELEGATE_URL` + `HANDOFF_DELEGATE_ASSISTANT_ID` (+ `HANDOFF_DELEGATE_REPLY_NODE`
   for voice) when delegating. `delegateInput` should forward the request context plus the
   session identity slots.
2. **Tool opt-in**: pass `handoff: handoffSpec` to `buildAgentStepTool` in the tool's
   `index.ts`.
3. **Graph wiring**: `createReactAgent` cannot express the conditional edge — hand-roll the
   graph: `addConditionalEdges("tools", s => handoffRequested(s) ? HANDOFF_NODE : "agent")`,
   `addNode(HANDOFF_NODE, createHandoffNode(handoffSpec))`, `addEdge(HANDOFF_NODE, END)`.
4. **Prompt**: a HANDOFF section — off-topic ⇒ sole `request_handoff` step with `context` in
   the customer's language, produce no answer text afterwards; borderline rule (mixed
   requests: answer the in-scope part, no handoff); SCOPE routes out-of-scope to the handoff
   instead of a blanket self-refusal.
5. **Clients/middleware**: stream `["messages-tuple", "custom"]` and follow checklist item 8
   of `streaming-and-channel-contract.md` (never route on `delegated_to`-only messages).
