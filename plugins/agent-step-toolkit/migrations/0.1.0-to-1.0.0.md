# Migrate agent-step 0.1.0 → 1.0.0

> Severity: breaking. Applied by `/pull-library`; readable standalone.

## What changed

The runner gained a **state selector** layer. Previously the runner handed every executor the
whole state and derived the executor's registry key from the action name by snake-to-camel. Now:

- Each action has a **selector** (`stateSelector.ts`, exporting `getSlice` + a `Slice` type) that
  projects the host state down to the slice that action's executor needs.
- The runner dispatches `executors[action](params, selectors[action](view))` — selectors and
  executors are both registered **1:1 under the exact action name** (snake_case). The snake-to-camel
  convention is gone.
- The executor receives the **slice**, not the whole state. Its `stateUpdate` may still patch any
  host slot.
- `buildAgentStepTool` / `runSteps` gained a required `selectors` option.

Before:
```ts
// index.ts
const executors: ExecutorRegistry<State> = { verifyCustomer, listAccounts };
buildAgentStepTool({ config, stateAnnotation, executors, verifiers });

// actions/verify_customer/executor.ts
export async function verifyCustomer(params: unknown, state: State): Promise<ExecutorResult<State>> { … }
```

After:
```ts
// actions/verify_customer/stateSelector.ts
export const getSlice = (s: State) => ({ userId: s.userId, customerCode: s.customerCode });
export type Slice = ReturnType<typeof getSlice>;

// index.ts
const selectors = {
  verify_customer: verifyCustomerSlice,
  list_accounts:   listAccountsSlice,
} satisfies SelectorRegistry<State, ActionName>;
const executors: ExecutorRegistry<State, typeof selectors> = {
  verify_customer: verifyCustomer,   // KEY is the action name; the fn name is free
  list_accounts:   listAccounts,
};
buildAgentStepTool({ config, stateAnnotation, selectors, executors, verifiers });

// actions/verify_customer/executor.ts
import type { Slice } from "./stateSelector.js";
export async function verifyCustomer(params: unknown, state: Slice): Promise<ExecutorResult<State>> { … }
```

Two bundled behaviour changes (no migration action needed): the runner now **catches executor
throws** and surfaces them as `ok:false` (`error: "executor_error"`) instead of letting them escape
`runSteps`, and the inert `proposedAt` timestamp was removed from pending confirmations.

## Why

Least-privilege state: an executor should declare (via its selector) exactly the slots it reads,
rather than receiving the whole graph state and reaching into anything. Keying selectors and
executors per-action lets `ExecutorRegistry<T, typeof selectors>` type each executor's `state` param
from its selector's return — so a selector/executor mismatch is a **compile error at the
`buildAgentStepTool` boundary**, not a runtime surprise. Dropping snake-to-camel removes a layer of
implicit naming magic: the registry key is just the action name.

## Library replacement

The vendored `src/agent-step/*` is replaced wholesale by `/pull-library` (verbatim copy from the
toolkit). `types.ts`, `runner.ts`, `runner.test.ts`, and `index.ts` changed; `state.ts` and
`define-config.ts` are unchanged. No library files were added or removed.

## Manual follow-ups

- **Narrow each selector.** Transform 3 ships identity selectors (`(s) => s`) so the project keeps
  compiling. Replace each with a real projection of just the slots that action's executor reads —
  that is the whole point of the selector layer. The toolkit's `state-selector.ts.template` and the
  `executor-patterns.md` reference show the intended shape; the source banking_info tool is a worked
  example (e.g. `list_accounts` selects only the session context).
- After narrowing, an executor that read a slot its selector no longer exposes will fail to compile —
  that's the design. Either add the slot to the selector (if the action legitimately needs it) or
  remove the stale read.
